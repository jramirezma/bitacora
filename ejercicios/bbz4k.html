El ejercicio

A) Instanciar 3 objetos de la clase Celular de 3 maneras y identificar al menos 4 atributos significativos.
B) Dado muchas falsificaciones de celulares, haz el método que verifique si un cel. es del mismo modelo que el otro y entonces igualar los atributos de el equipo modificado.
C) Sobrecargar el método mostrar.

Antes de codear

Pensá en tres celulares sobre una mesa. El primero no tiene caja ni datos — apareció con valores vacíos. El segundo tiene modelo y precio pero le faltan datos de fábrica. El tercero viene completo: modelo, marca, precio y almacenamiento. Esas tres situaciones son los tres constructores del inciso A.

La sobrecarga de constructores permite crear objetos del mismo tipo con distinta cantidad de información inicial; la sobrecarga de métodos permite que el mismo nombre de operación haga cosas ligeramente distintas según los argumentos que recibe.

Para el inciso B, la lógica es: "si este celular tiene el mismo modelo que aquél, copiá sus atributos". No creás un objeto nuevo; modificás el objeto receptor usando los setters del objeto que se pasa como parámetro. Para el inciso C, mostrar() sin argumentos imprime los datos propios, y mostrar(String propietario) agrega quién es el dueño — mismo nombre, distinta firma.

Un celular falso con los atributos del original: básicamente el inciso B resuelve el problema que ningún perito de Mercado Libre pudo.

Algo que vale la pena saber

La sobrecarga de métodos (method overloading) existe en los lenguajes desde antes de que la orientación a objetos se popularizara, pero fue la POO la que la convirtió en una herramienta de diseño cotidiana. El compilador distingue qué versión del método llamar según la cantidad y el tipo de los argumentos, no según el nombre. Por eso dos métodos mostrar() y mostrar(String propietario) conviven sin conflicto: son firmas distintas aunque compartan nombre.

En Java, el constructor también es un método sobrecargado. Cada vez que escribís new Celular(), new Celular("X", 999) o new Celular("X", "Samsung", 999, 128), el compilador elige cuál de los tres bloques ejecutar basándose únicamente en los argumentos que le pasás.

El diseño

El ejercicio se modela con una sola clase Celular. Tiene cuatro atributos privados que representan las características del equipo. Expone tres constructores sobrecargados, getters y setters para cada atributo, el método Verificar que recibe otro objeto Celular, y dos versiones sobrecargadas de mostrar.

CLASE Celular 
// Constructor por defecto
Celular() {
    self.modelo         = ""
    self.marca          = ""
    self.precio         = 0
    self.almacenamiento = 0
}

// Constructor con modelo y precio
Celular(String modelo, float precio) {
    self.modelo         = modelo
    self.marca          = ""
    self.precio         = precio
    self.almacenamiento = 0
}

// Constructor completo
Celular(String modelo, String marca, float precio, int almacenamiento) {
    self.modelo         = modelo
    self.marca          = marca
    self.precio         = precio
    self.almacenamiento = almacenamiento
}

// Getters
String getModelo()         { retornar modelo }
String getMarca()          { retornar marca }
float  getPrecio()         { retornar precio }
int    getAlmacenamiento() { retornar almacenamiento }

// Setters
void setModelo(String m)      { self.modelo = m }
void setMarca(String m)       { self.marca = m }
void setPrecio(float p)       { self.precio = p }
void setAlmacenamiento(int a) { self.almacenamiento = a }

// Inciso B — verifica modelo y copia atributos si coincide
void Verificar(Celular b) {
    SI self.modelo == b.getModelo()
        self.setMarca(b.getMarca())
        self.setPrecio(b.getPrecio())
        self.setAlmacenamiento(b.getAlmacenamiento())
        IMPRIMIR "LOS MODELOS SON IGUALES"
    SINO
        IMPRIMIR "LOS MODELOS NO SON IGUALES"
}

// Inciso C — mostrar sin argumentos
void mostrar() {
    IMPRIMIR modelo + " " + marca + " " + precio + " " + almacenamiento
}

// Inciso C — mostrar con propietario (sobrecarga)
void mostrar(String propietario) {
    IMPRIMIR "EL CELULAR DE MODELO " + modelo + " LE PERTENECE A " + propietario
}

Tracemos a mano

Creamos los tres objetos del inciso A:
a → modelo="", marca="", precio=0, almacenamiento=0
b → modelo="AX24", marca="", precio=6569.32, almacenamiento=0
c → modelo="AX24", marca="SAMSUNG", precio=7000.99, almacenamiento=100

Las tres llamadas a mostrar() imprimen esos valores tal cual.

Inciso B: c.Verificar(b). Se compara c.modelo ("AX24") con b.getModelo() ("AX24") — son iguales. Entonces c copia los atributos de b sobre sí mismo:
c queda → modelo="AX24", marca="", precio=6569.32, almacenamiento=0
Se imprime: "LOS MODELOS SON IGUALES".

Inciso C: c.mostrar() imprime los 4 atributos actuales de c. c.mostrar("JOSUE") imprime "EL CELULAR DE MODELO AX24 LE PERTENECE A JOSUE". Mismo nombre, dos comportamientos distintos según la firma invocada.

Resultado. Se modeló la clase Celular con cuatro atributos privados (modelo, marca, precio, almacenamiento), tres constructores sobrecargados para distintos niveles de inicialización, getters y setters completos, el método Verificar(Celular b) que compara modelos y copia atributos si coinciden, y dos versiones sobrecargadas de mostrar: una que imprime los datos del equipo y otra que agrega el nombre del propietario.

¿Ya usé esto sin saberlo?

Cada vez que en Java escribís System.out.println(42), System.out.println("hola") o System.out.println(3.14), estás usando sobrecarga de métodos. println tiene más de diez versiones en la clase PrintStream, una por cada tipo primitivo y una para objetos. El compilador elige cuál ejecutar según lo que le pasás — vos nunca lo notás porque el nombre siempre es el mismo.

La sobrecarga de constructores aparece en clases de uso cotidiano como StringBuilder, que puede construirse vacío, con una cadena inicial, o con una capacidad inicial. Mismo patrón que el Celular de este ejercicio: distintas firmas para distintas situaciones de arranque.
