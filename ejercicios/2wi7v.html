Lo que nos piden

Crea un programa donde pueda instanciar objetos de la clase Persona con los atributos (nombre, edad) y un método que me compare la edad de una persona con la de otras dos personas y me muestre quién es el mayor imprimiendo los datos de esa persona.

Entendamos el problema

Imaginá que tenés tres personas en una fila. Querés saber cuál es la de mayor edad. Pero hay un detalle: el método Comparar vive dentro de uno de los objetos — digamos a — y recibe a los otros dos (b y c) como parámetros. Entonces a actúa como árbitro: compara su propia edad con la de b y con la de c para determinar quién gana.

Un método de instancia puede recibir otros objetos de su misma clase como parámetros y acceder a sus atributos a través de los getters. Así un objeto "habla" con otro sin romper el encapsulamiento.

Los casos posibles son cuatro: las tres edades son iguales, solo a es la mayor, solo b es la mayor, solo c es la mayor, o hay empate entre dos. La cadena de SI / SINO SI maneja cada uno en orden.

Tres personas, una sola mayor. Si vivieran en un reality show, este método decidiría quién sigue y quién se va a casa.

Un dato que cambia todo

Cuando un método de una clase recibe como parámetro otro objeto de esa misma clase, en POO se lo llama a veces "método de comparación entre pares" (peer comparison). Este patrón es la base de cómo funciona la interfaz Comparable en Java: su único método, compareTo(T otro), recibe un objeto del mismo tipo y devuelve un entero que indica el orden relativo. Toda la maquinaria de ordenamiento de Java — Collections.sort, Arrays.sort, los árboles ordenados — depende de ese único método que sigue exactamente la misma idea que Comparar(Persona x, Persona y).

Manos a la obra

Una sola clase Persona con dos atributos privados. El método Comparar recibe dos objetos Persona y determina cuál de los tres — el receptor o cualquiera de los dos parámetros — tiene la mayor edad.

Persona
- String nombre
- int edad
+ leer()
+ mostrar()
+ String getNombre()
+ int getEdad()
+ void Comparar(Persona x, Persona y)

leer() carga los datos del objeto por teclado. mostrar() los imprime. Los getters permiten que Comparar acceda a la edad de x e y sin tocar directamente sus atributos privados. El objeto que invoca Comparar usa su propio atributo edad directamente, porque está dentro de la clase.

CLASE Persona 
// Carga datos por teclado
void leer() {
    leer(nombre, edad)
}

// Imprime los datos de esta persona
void mostrar() {
    IMPRIMIR nombre + " " + edad
}

// Getters
String getNombre() { retornar nombre }
int    getEdad()   { retornar edad }

// Compara la edad de self con la de x e y
// y muestra los datos del mayor
void Comparar(Persona x, Persona y) {
    int ea = edad
    int ex = x.getEdad()
    int ey = y.getEdad()

    SI (ea == ex) Y (ea == ey)
        IMPRIMIR "Las tres personas tienen la misma edad"
    SINO SI (ea < ex) Y (ea < ey)
        SI ex > ey
            x.mostrar()
        SINO SI ey > ex
            y.mostrar()
        SINO
            IMPRIMIR "Existen dos personas con la mayor edad"
    SINO SI (ex < ea) Y (ex < ey)
        SI ea > ey
            self.mostrar()
        SINO SI ey > ea
            y.mostrar()
        SINO
            IMPRIMIR "Existen dos personas con la mayor edad"
    SINO SI (ey < ea) Y (ey < ex)
        SI ea > ex
            self.mostrar()
        SINO SI ex > ea
            x.mostrar()
        SINO
            IMPRIMIR "Existen dos personas con la mayor edad"
    SINO
        IMPRIMIR "Existen dos personas con la mayor edad"
}
  Lee nombre y edad desde teclado       Muestra nombre y edad del objeto receptor     Devuelven el atributo correspondiente         ea = edad del objeto que llama al método ex = edad de x, accedida por getter ey = edad de y, accedida por getter   Caso 1: las tres son iguales   Caso 2: a es la menor → mayor es x o y x gana entre x e y   y gana entre x e y   x e y empatan como máximas Caso 3: x es la menor → mayor es a o y a gana entre a e y   y gana entre a e y   a e y empatan como máximas Caso 4: y es la menor → mayor es a o x a gana entre a e x   x gana entre a e x   a y x empatan como máximas Caso residual: empate no cubierto arriba  
PROGRAMA PRINCIPAL 
// Crear tres objetos Persona
Persona a = nuevo Persona()
Persona b = nuevo Persona()
Persona c = nuevo Persona()

// Cargar datos por teclado
a.leer()
b.leer()
c.leer()

// a actúa como árbitro y compara las tres edades
a.Comparar(b, c)
  Tres instancias independientes       Cada objeto recibe su nombre y edad       Dentro de Comparar, a usa su edad directamente y accede a b y c a través de sus getters

Probemos con un ejemplo

Datos ingresados:
a → nombre="Ana", edad=30
b → nombre="Luis", edad=45
c → nombre="Marta", edad=38

Se llama a.Comparar(b, c). Dentro del método: ea=30, ex=45, ey=38.
¿(30==45) Y (30==38)? No.
¿(30<45) Y (30<38)? Sí — a es la menor.
¿ex > ey? ¿45 > 38? Sí → se imprime b.mostrar().
Salida: "Luis 45".

Caso de empate: a=30, b=45, c=45.
ea=30, ex=45, ey=45. Segunda condición se cumple.
¿ex > ey? No. ¿ey > ex? No → salida: "Existen dos personas con la mayor edad".

Resultado. Se modeló la clase Persona con atributos privados nombre y edad, métodos leer() y mostrar(), getters para ambos atributos, y el método Comparar(Persona x, Persona y) que evalúa los cuatro casos posibles — igualdad total, cada una como única mayor, y empate entre dos — para imprimir los datos de la persona con mayor edad o informar si hay empate.

¿Dónde ya existe esto?

El patrón de recibir otro objeto de la misma clase como parámetro para compararlo es exactamente lo que hace la interfaz Comparable en Java. Cuando una clase implementa compareTo(T otro), le está diciendo al lenguaje cómo ordenarse respecto de otro objeto del mismo tipo. Cada vez que usás Collections.sort sobre una lista de objetos, por debajo se invocan llamadas a ese método — el mismo mecanismo que Comparar, reducido a retornar un entero negativo, cero o positivo en lugar de imprimir un mensaje.

La diferencia de diseño es interesante: Comparar imprime el resultado directamente (responsabilidad mezclada), mientras que compareTo solo devuelve un valor y deja que quien llama decida qué hacer con él. Esa separación — calcular vs. actuar — es uno de los principios que aparece constantemente a medida que los programas crecen.
